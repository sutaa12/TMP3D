Unity / TextMeshPro / Compute Shader

TextMeshProを 3D文字として 描く仕組み

このプロジェクトの CustomUnlit_Compute は、 TMP文字を厚みのある形へ変換します。 SDFでピクセル単位に中身を判定し、 処理を「データの旅」として 追えるように分解しました。

TMPは文字ごとに四角形を作ります。ここでは “TMP” の3文字なので、入力は3クワッド、つまり12頂点です。

TextMeshPro SDF Basics

ぼけた画像なのに、なぜ文字がきれいに出るのか

TMPのSDFアトラスは、文字をそのまま写真のように保存している画像ではありません。 各ピクセルに「文字の輪郭までどれくらい近いか」をグレー値として入れた距離マップです。 だから見た目はぼんやりしていても、シェーダー側で「どこを輪郭にするか」を毎回計算し直せます。

SDFは「ぼけ画像」ではなく「距離のメモ」

左のSDF画像はぼやけて見えますが、真ん中のしきい値で輪郭を決め直すことで、右のようなシャープな文字になります。

01 / 保存しているもの

白黒の濃さは「距離」

白に近いほど文字の内側、黒に近いほど外側、真ん中のグレー付近が輪郭です。 通常画像のような「このピクセルを何色で塗るか」ではなく、輪郭からの距離を圧縮して持っています。

02 / 描くとき

シェーダーが境界を切り出す

描画時にSDF値を読み、だいたい `0.5` を輪郭として扱います。 さらに `smoothstep` のような補間で境界幅を調整するので、拡大縮小しても輪郭がギザギザしにくくなります。

03 / 3D化で効く理由

奥行き方向でも同じ輪郭を使える

今回の3D文字では、箱の中の位置をSDFアトラスのUVへ戻して「ここは文字の中か」を判定します。 つまり、前面だけでなく奥行き面も同じ距離情報で形を決められます。

SDF alpha 輪郭までの距離っぽい値
edge = 0.5 この近辺を文字の境界にする
screen scale 画面上の大きさに合わせて境界幅を調整
smooth edge 結果としてシャープで滑らかな文字になる

Pipeline

全体像は「板を作る、厚みを作る、画素を決める」

C#、compute shader、描画シェーダーは同じ仕事をしているのではなく、きれいに役割分担しています。 まずCPU側でTMPのメッシュを読み、GPU側で押し出し用バッファを作り、最後にピクセルシェーダーがSDFを見ながら塗ります。

01

TMP Mesh

TextMeshPro が文字ごとに4頂点のクワッドを生成します。UVはSDFアトラスの位置を指します。

02

C# Pack

`TMP3D_ComputeRenderer` が頂点、UV、色、文字のdepthを `InVertex` としてまとめます。

03

Compute

`TMP3D_Extrude.compute` が1クワッドを16頂点と36インデックスへ展開します。

04

Shader

描画時に `DrawProceduralNow` でバッファを読み、レイマーチとSDFで文字の表面を決めます。

Extrusion

1文字の平面クワッドを、箱のような描画範囲へ押し出す

TMPの文字は、見た目が複雑でもメッシュとしては基本的に四角形です。 compute shaderはその四角形を前面、背面、側面に分けて、3D空間でサンプルできる箱を作ります。

  • 入力: 4頂点、UV、色、depth
  • 出力: front 4頂点、back 4頂点、side 8頂点
  • インデックス: 12三角形分、合計36個

SDF + Raymarch

立体の中を少しずつ進み、文字の面に当たったら色を置く

SDFは「そのUVが文字の内側にどれくらい近いか」をグレースケールで持つテクスチャです。 ピクセルシェーダーは視線方向に小さく進み、現在位置を3Dマスク座標へ変換し、SDFの値がしきい値を越えたところを表面として扱います。

SDFアトラスの考え方

白に近いほど文字の内側、黒に近いほど外側です。輪郭は0.5付近の境界として扱えます。

レイマーチの進み方

箱の中を視線方向に進み、SDFが文字の面を示す場所でヒットします。step数を増やすほど細かく探索します。

Mental Model

コードを読むときの入口

初心者が迷いやすいのは「どこで本物の3Dメッシュを作っているのか」です。 この実装では、CPUでUnityのMesh Assetを作るのではなく、GPUバッファとシェーダーで描画時に組み立てています。 

TMP text changed
  -> read textInfo.meshInfo
  -> collect visible character quads
  -> pack position, uv, color, depth
Compute shader
  -> one thread handles one quad
  -> create front, back, side vertices
  -> write vertex and index buffers
Render shader
  -> fetch buffer by SV_VertexID
  -> raymarch inside character box
  -> sample SDF atlas and shade hit

Constraints

普通のTextMeshProと違うところ

この仕組みはTextMeshProの文字情報を使いますが、通常のTMPと同じ感覚でフォントやマテリアルを差し替えられるわけではありません。 「文字の板」を3Dサンプル用の箱として扱うため、SDFアトラス、カスタムシェーダー、GPU機能に依存します。

Font atlas

SDFフォントが前提

レイマーチ中に `_MainTex` のSDF値を読むため、輪郭情報を持つTMP SDFフォントが必要です。 Bitmapフォントや解像度の低いアトラスでは、立体の輪郭や奥行き面が荒く見えます。

Material

通常のTMPマテリアルだけでは完結しない

`_FaceTex`、`_DepthAlbedo`、`_OutlineTex`、レイマーチ系パラメータなど、3D用の追加プロパティを使います。 新しいフォントを使う時は、フォントアセットだけでなく対応マテリアルも確認します。

Component

UIのTMPとは別物として考える

現在の `TMP3D_ComputeRenderer` は `TextMeshPro` コンポーネントを要求します。 `TextMeshProUGUI` をCanvas上でそのまま3D化する用途ではなく、ワールド空間のTMPを描く想定です。

Glyph features

リッチな文字機能は検証が必要

基本は「可視文字の4頂点クワッド」を処理します。fallbackフォント、絵文字、sprite glyph、underline、strikethrough、 複数マテリアル文字は、通常TMPと同じ見え方になるか個別確認が必要です。

Platform

Compute shaderとShader Model 5.0に依存

compute版はGPUバッファと `DrawProceduralNow`、描画シェーダーの `#pragma target 5.0` を使います。 WebGLや一部モバイルなど、compute対応が弱い環境では別方式やフォールバックを検討します。

Rendering

本物のメッシュ資産ではない

見た目は3Dですが、Unity上に編集可能な押し出しMeshを作っているわけではありません。 MeshCollider、メッシュ編集、ライトマップ用ジオメトリのような用途には向きません。

Polygon vs Raymarching

ポリゴン化とレイマーチング、どちらが楽か

3D文字には大きく分けて、文字の輪郭から本物のポリゴンを生成する方法と、この実装のように箱の中をSDFで探索する方法があります。 手軽さ、処理負荷、見た目の自由度はトレードオフです。

2つの考え方

左は輪郭そのものを三角形化する考え方、右は押し出し箱の中でSDFを読む考え方です。

ポリゴン押し出しが向く場面

  • 文字形状を本物のMeshとして扱いたい
  • 物理、影、ライト、メッシュ編集が必要
  • 静的なロゴや少数の文字を高品質に作り込みたい

今回の方式が向く場面

  • TMPのテキスト更新に追従しやすくしたい
  • フォント輪郭の三角形化を避けたい
  • Depth色、アウトライン、SDF由来の輪郭表現をシェーダーで調整したい
観点 ポリゴン化した3D文字 今回のレイマーチング方式
導入の手軽さ フォント輪郭の取得、三角形分割、側面生成が必要。動的文字では実装量が増える。 TMPの4頂点クワッドとSDFアトラスを利用できる。フォント輪郭の直接処理を避けられる。
処理負荷 頂点数と三角形数が文字の複雑さに比例。生成時CPU負荷、描画時頂点負荷が中心。 形状は軽めだが、ピクセルごとに複数stepのSDFサンプルを行う。画面上で大きい文字ほど重くなりやすい。
見た目の品質 シルエットはポリゴン密度と三角形分割品質に依存。近距離でも実体感を出しやすい。 SDFとstep数に依存。輪郭、depth色、アウトラインをシェーダーで変えやすい。
Unity機能との相性 MeshRenderer、MeshCollider、影、ライト、ベイクなど標準機能に乗せやすい。 描画専用に近い。Colliderや編集可能Meshが必要な用途には追加実装が必要。
動的テキスト 文字変更ごとにメッシュ再生成が必要になりやすい。 TMPの変更イベントからGPUバッファを作り直せるため、カウントアップなどの演出と相性が良い。
プラットフォーム 通常メッシュなら対応範囲は広め。ただし生成コードやデータ量に注意。 compute shader、StructuredBuffer、Shader Model 5.0に依存。対象プラットフォーム確認が重要。

Glossary

グラフィック初心者向けの用語

SDF

Signed Distance Fieldの略です。文字の輪郭を、ただの白黒画像ではなく「境界からの距離っぽい値」として持つので、拡大しても輪郭をきれいに出せます。

Compute Shader

頂点やピクセルを直接描く前に、GPUで汎用計算をする仕組みです。このサイトの例では、文字ごとの押し出し頂点をまとめて作ります。

Raymarching

光線を少しずつ前へ進めて、形に当たった場所を探す描き方です。三角形の表面だけでなく、ボリューム内の条件を見ながら表面を決められます。

GraphicsBuffer

GPUが読むための配列です。C#側で作った `GraphicsBuffer` をMaterialへ渡すことで、シェーダー内の `StructuredBuffer` から参照できます。
